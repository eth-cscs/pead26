# Read the Docs: Best Practices for Site Documentation (BoF)

Sites provide sophisticated technical solutions and services that require a high level of technical competence to use.
Technical user-facing documentation is an essential part of these services, however the time and skills required to write and maintain quality documentation are easy to underestimate for both technical staff and management.

The documentation at sites including NERSC, CSCS and ORNL has been developed bottom-up by engineers trying to address the needs of users.
This BoF will foster an exchange of best practices and ideas for user-facing documentation, based on the experiences of these engineers who maintain documentation infrastructure and write the documentation.

## Audience participation

Present

1. Short presentations from sites about their documentation strategy;
2. A panel discussion with the questions from the audience, and a list of backup talking points to drive the conversation;
3. The audience to submit examples of the and the good in their own docs using an online tool:
    - show submissions on the screen, and invite the panel and audience to comment after the submitter from the audience has briefly introduced it.

In order to keep the conversation focussed, the presenters will be asked choose two or more of the following topics:

Show and tell one part of your documentation that you are particularly proud of.
Which framework do you use for documentation deployment (CMS or docs-as-code)?
How do you organise documentation to make information discoverable by users and maintainable by writers.
How is responsibility distributed? For example is there a core documentation team, or is the workload distributed over all engineering teams?
Lessons learnt implementing LLM based chatbot services for users.
Have we been able to get contributions from the user community?

These themes will be introduced at the start of the BoF when the audience is asked to submit examples from their site's documentation, and used as guiding topics during the panel discussion after the presentations.
Schedule
The session is designed to run for between 60-90 minutes, with 90 minutes preferred.
The timings given below are based on 90 minutes, and will be adjusted for a smaller time slot.

10 minutes: Introduction. Wait for the audience to arrive and a brief presentation by the session chair introducing the topic and the themes, and asking the audience to start submitting their examples.

30 minutes: Site presentations. The length of each presentation will be determined by the number of sites. We have 3 sites committed to present (ORNL, Pawsey and CSCS), and will seek more presenters through direct invitation and an open call on the CUG Slack.

45 minutes: Panel discussion. 
Field questions and topics from the audience;
Discuss "show and tell" examples provided by the audience: the chairperson will also encourage audience input
The chairperson will have their own list of questions and examples to use to stimulate audience participation if needed.

5 minutes: Wrap up and closing words. Can be skipped to make room for constructive panel discussion).
